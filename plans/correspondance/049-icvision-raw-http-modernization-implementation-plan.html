Request Summary

The PI requested a deliberately detailed implementation plan that decomposes the raw HTTP modernization into small, reversible steps and accounts for compatibility, security, failure behavior, edge cases, testing, comparison evidence, rollback, and eventual SDK retirement.

Desired outcome: a buildable sequence that removes the SDK safely without losing the validated institutional endpoint path or corrupting ICVision results. Non-goals: no implementation, dependency removal, credential inspection, provider call, patient-data publication, scientific equivalence claim, merge, or release in this record.

Constraints: one sole editor per increment; independent security, code, and QA checks; no automatic cross-transport fallback; no arbitrary credential destination in the raw lane; no all-subject validation; no path-hygiene or environment-setup side work unless it directly blocks an approved increment.

Recommended Architecture

Use a temporary hybrid migration. Preserve the SDK as the existing compatibility and rollback lane while raw HTTP is introduced as an explicit opt-in lane. Raw HTTP may use only named, reviewed endpoint profiles. It must never accept or silently translate an arbitrary base_url, inherit ambient endpoint or proxy routing, follow a redirect, or automatically fall back to the SDK.

An approved profile owns the exact HTTPS host, port, Responses path, TLS policy, credential source, allowed request settings, and resource limits. The institutional route is represented as an explicit profile, not an ambient HTTP proxy. The SDK rollback lane remains available until raw HTTP has passed private comparison, independent review, and an explicit default-switch decision.

Required PI decision before implementation: approve or modify this temporary hybrid/profile-based endpoint and credential policy. Later decisions about retries, raw becoming the default, and SDK removal remain separate.

Goal and Invariants

Goal: replace the SDK request boundary with a small project-owned HTTPS implementation while preserving classification behavior, endpoint compatibility, result structure, privacy, and comparison evidence.

  • Every Responses request explicitly contains store: false.
  • Raw HTTP cannot send a credential to a caller-selected destination.
  • No redirect or ambient proxy can change the destination.
  • Credentials, prompts, images, Base64, provider bodies, private paths, and protected reasons stay out of logs, Git, and public records.
  • A transport or parser failure produces confidence 0.0, exclude_vision=False, and no data mutation.
  • The first raw live proof is one representative component, one request, one connection, concurrency one, and no automatic retry or fallback.
  • Software smoke evidence is not scientific-equivalence evidence.
  • Implementation quality is measured through behavior and evidence rather than worktree or directory tidiness.

Highest-Learning Question

Can the raw transport reproduce the SDK lane's application-facing behavior with the same protected derived image, prompt, model, endpoint profile, and parser while proving that the request goes only to the reviewed endpoint, is not retried, cannot auto-exclude on failure, and leaks no protected material?

Implementation Ladder

Gate 0 - Compatibility and credential policy

  1. Authorize the SDK lane to retain legacy custom-endpoint compatibility temporarily.
  2. Authorize immutable named profiles for the raw lane.
  3. Bind each raw profile to a reviewed credential source after profile validation.
  4. Keep the public command-line key compatibility question under PI custody; prefer environment-only credentials for governed raw live use.
  5. Make raw plus base_url fail before data loading, plotting, or connection creation.
  6. Ignore ambient endpoint and proxy variables in the raw lane.
  7. Require a separate review if a private certificate authority is ever needed; never disable certificate verification.

Increment 1 - Freeze the offline wire and compatibility contract

  1. Use only a synthetic prompt and synthetic image fixture.
  2. Capture the SDK's serialized request through a fake HTTP transport with no socket.
  3. Compare model, prompt, image data URL, input ordering, temperature, reasoning, image detail, output bound, and storage setting.
  4. Resolve the current two-message input shape versus the candidate's combined content array.
  5. Label intentional safety changes explicitly: store:false, bounded bodies, bounded output, sanitized failures, and no retry.
  6. Define the minimal internal transport result: output text, normalized usage if available, and a stable sanitized outcome category.
  7. Freeze the immutable endpoint-profile structure and reject unknown or URL-shaped profile values before credential resolution.

Acceptance: no real socket; SDK and raw fixtures differ only for approved safety deltas; marker secrets and protected content never appear in snapshots, logs, exceptions, or artifacts.

Increment 2 - Harden the inert raw transport

  1. Use candidate commit be78c8f only as seed code.
  2. Resolve the fixed profile before reading credentials.
  3. Expose no caller-supplied host, path, port, proxy, SSL context, opener, redirect destination, or arbitrary headers.
  4. Construct a closed header set: authorization, JSON content and acceptance, identity encoding, and an optional fixed user agent.
  5. Reject CR, LF, NUL, TAB, DEL, and other ASCII controls in header-bound values.
  6. Use the platform trust store, hostname verification, certificate-required TLS, and the exact profile hostname for SNI.
  7. Reject every redirect without following Location.
  8. Use the existing Pillow dependency to verify decoded image format, extension/MIME agreement, dimensions, pixel count, truncation, and corruption.
  9. Read one opened file descriptor with a limit-plus-one strategy.
  10. Bound image bytes, prompt contribution, request bytes, response bytes, dimensions, pixels, output tokens, socket timeout, and overall deadline.
  11. Require compact UTF-8 JSON containing store:false.
  12. Use one fresh connection and close it in finally.
  13. Require the expected success status, JSON media type, and identity content encoding.
  14. Reject oversized declared bodies before reading and bound chunked or unknown-length bodies during reading.
  15. Never retain a non-success provider body; return only stable error categories.
  16. Perform no retry.

Acceptance: below/at/above limit tests; valid and invalid image formats; hostile routing variables; TLS, redirect, HTTP-status, encoding, malformed-response, redaction, and connection-cleanup tests; no production import yet.

Increment 3 - Strict parser and safe-result adapter

  1. Separate HTTP mechanics from classification parsing.
  2. Allow reasoning items before the message but require exactly one message and one nonempty output-text value for the initial contract.
  3. Reject incomplete, failed, cancelled, missing, or unknown Responses states.
  4. Reject malformed or ambiguous JSON, duplicate keys, missing or unknown labels, wrong types, Boolean confidence, NaN, infinity, out-of-range confidence, unexpected fields, and oversized or control-bearing reasons.
  5. Represent operational failure separately from a legitimate artifact classification.
  6. Project failures to the established output boundary with confidence 0.0, a stable redacted reason, and exclude_vision=False.
  7. Keep provider explanations protected and bounded if retained locally; never log or publicly publish them.
  8. Track attempts, successful parsed responses, and usage-bearing responses separately.
  9. Leave missing usage unavailable and unknown-model cost unknown; remove silent fallback pricing.

Acceptance: frozen response-state and classification fixtures; unchanged public return signatures and DataFrame structure; no ICA exclusion or data mutation after any injected failure.

Increment 4 - Opt-in single-layout integration

  1. Add a narrow explicit transport selector and named endpoint-profile selection through CLI, core, compatibility, and API layers.
  2. Validate every transport/profile/base-URL combination before reading research data or generating plots.
  3. Keep SDK behavior isolated in the SDK lane and raw behavior isolated in the raw lane.
  4. Never auto-fallback between lanes.
  5. Route only single layout through raw HTTP; raw plus strip fails during preflight.
  6. Preserve prompt selection, label mapping, confidence behavior, result order, and DataFrame structure.
  7. Do not silently change model defaults, reasoning defaults, strip defaults, structured-output behavior, or prompts during transport migration.
  8. Pin the first raw execution contract to one component, one attempt, one active connection, no auto-exclusion, and no report.
  9. Add a fully synthetic end-to-end test from application entry point through the fake raw connection and result DataFrame.
  10. Record the selected lane and logical profile in a sanitized private manifest.

Acceptance: focused transport/integration tests, affected API and CLI regressions, full suite, diff check, and proof that neither lane calls the other.

Increment 5 - Private paired one-component comparison

  1. Obtain separate PI authorization for one live raw request.
  2. Use only the PI-approved representative component and keep every input and derivative private.
  3. Use the exact retained derived image for both lanes when available; otherwise generate once and use the same verified image for both.
  4. Hold image hash, prompt hash, model, image detail, endpoint profile, output bound, parser, and exclusion/report settings constant.
  5. Verify the exact profile destination, hostname/certificate checks, no redirect, no ambient proxy, and no header substitution.
  6. Enforce one request with no transport retry, redirect retry, application retry, or cross-transport fallback.
  7. Store a private sanitized manifest with code revision, logical lane/profile, protected hashes, settings, store:false, attempt count, elapsed time, status, normalized classification, exclusion flag, and usage availability.
  8. Never retain credentials, authorization headers, Base64, full request bodies, raw provider bodies, environment dumps, or sensitive paths in publishable material.
  9. Compare structural and operational behavior, not exact label equality or scientific accuracy.

Acceptance: valid label taxonomy, finite confidence, stable schema, safe failure semantics, truthful usage/cost representation, verified routing, and no protected-material leak.

Increment 6 - Strip, batching, retry policy, and concurrency

  1. Migrate strip transport and strip parser independently from the proven single path.
  2. Preserve component-letter mapping, partial final strips, one result row per expected component, and deterministic ordering.
  3. Remove raw provider-response logging.
  4. Define safe failures for missing, duplicated, or mismatched component IDs.
  5. Start with one attempt and concurrency one.
  6. Use synthetic barriers to prove active requests never exceed the configured concurrency and components are neither lost nor duplicated.
  7. Verify temporary-image cleanup after plotting, worker, HTTP, parser, timeout, and cancellation failures.
  8. Do not inherit the existing broad three-attempt retry policy automatically.
  9. Require a later PI decision before any retry, including eligible statuses, ambiguous POST handling, duplicate cost, maximum attempts, backoff, jitter, cancellation, and final-attempt behavior.

Increment 7 - Raw HTTP becomes the default

  1. Require successful single and strip regression evidence and the private paired comparison.
  2. Require Cipher, The Boulder, and Cricket to review or test the exact implementation revision.
  3. Require both SDK and raw lanes to remain runnable through offline contract tests.
  4. Require an explicit, auditable selector that is never chosen after an error.
  5. Document operator rollback and record the selected lane in private manifests.
  6. Obtain explicit PI authorization for the default switch.
  7. Retain the SDK rollback lane for one tagged release and at least 30 days, with Raven as retirement owner.

Increment 8 - Retire the SDK

  1. Obtain the final PI decision on arbitrary custom-endpoint compatibility.
  2. Remove SDK request construction and SDK-specific exception handling.
  3. Remove the direct OpenAI package declaration from project, tox, and lockfile surfaces; regenerate rather than hand-edit the lockfile.
  4. Confirm whether the package remains transitively installed before claiming complete removal.
  5. Reconcile active scripts and operator guides while preserving historical correspondence.
  6. Build and install from the lockfile, run the complete suite, and search for active SDK imports/calls.
  7. Preserve Git history and the protected baseline bundle as migration readback evidence.

Conflict Status

Flag 1: Echo proposed strip-first integration because the strip helper already consumes text; Cricket proposed single-first because the established representative baseline is single-component.

Respond: strip-first has a smaller helper seam, but it immediately combines transport migration with strip parsing, broad retries, component mapping, and raw-response logging.

Decide: Raven selected single-first for the plan because it maximizes comparison value and isolates fewer behaviors. Strip remains a later increment.

Flag 2: Echo proposed commit-only rollback and no runtime selector; the comparison and cutover plan require both lanes to be explicitly runnable.

Decide: use a narrow temporary selector with no automatic fallback and a concrete retirement window.

Flag 3: Cipher recommended environment-only raw credentials, while existing public direct-key compatibility is still part of the product surface.

Decide: raw-live custody should be environment-bound; the public compatibility disposition remains with the PI.

Document: The Boulder approved the staged direction after requiring explicit profile-bound proxy/TLS rules, a closed header allowlist, an operable rollback-selector contract, and frozen parser/result fixtures. Those four requirements are included above.

Execution Decision

Disposition: Modify and stage. Accept the profile-based raw transport direction, but do not treat candidate be78c8f as merge-ready and do not remove the SDK in the first implementation. The recommended first implementation lock, after the PI compatibility decision, is the offline wire and compatibility contract. It provides the highest-learning evidence without touching credentials, research data, or a provider.
