Request Summary

Desired outcome: replace earlier Gate 0/1 work with a small, test-first foundation for a future project-owned Responses transport. The immediate goal was to settle compatibility and credential-routing policy, then freeze a synthetic request/result contract without connecting to any provider.

Non-goals: this record does not enable raw HTTP in the production pipeline, transmit EEG-derived content, read a credential, test a gateway, select a production model, remove the SDK lane, or claim scientific improvement.

Constraints: retain legacy SDK compatibility temporarily; keep raw endpoint selection profile-bound rather than caller-controlled; use only synthetic prompt and image fixtures; keep credentials, protected data, provider output, and private infrastructure details out of the public record.

Execution Update

Gate 0 and Gate 1 were replaced on the modernization branch and independently reviewed. The change adds a pure endpoint-policy boundary, a synthetic-only request and result contract, focused tests, and recurring Make targets. It deliberately does not add a production raw-client call path.

What is complete: the project now has an auditable offline contract for what a future raw Responses request must look like and which safety differences from legacy SDK behavior are intentional.

Gate 0 — Compatibility and Credential Policy

  • The SDK remains the default compatibility lane, including its existing custom-endpoint behavior.
  • The future raw lane accepts only an immutable reviewed profile. Unknown profile names, URL-shaped profile values, raw custom-base-URL inputs, and unsupported transports fail before credential resolution or pipeline work.
  • Ambient endpoint, proxy, and provider-key variables do not choose or alter the raw destination. Custom certificate-authority overrides fail closed; certificate verification is not disabled.
  • Raw remains unavailable from the public core and command-line path at this gate. The policy module creates no client, makes no socket, and resolves no credential.

This separates an important compatibility issue: preserving the SDK’s legacy flexibility does not silently authorize arbitrary credential routing in a new raw transport.

Compatibility Characterization

The current SDK serialization was captured locally through a fake HTTP transport with real sockets and sleeping denied. The characterization found that the legacy path sends two ordered user inputs and retains a temperature setting. Gate 1 preserves the model, prompt, image bytes, MIME, and semantic ordering while explicitly recording—not hiding—the intentional safety changes.

This is a characterization contract, not a claim of byte-for-byte wire parity or gateway compatibility.

Scope Limits

No live endpoint or model inventory was queried. No credential, secret store, raw or derived EEG data, participant identifier, provider response, local machine path, private gateway hostname, cost record, or deployment setting was inspected or published. Passing offline tests does not establish provider compatibility, retention behavior, model support, performance, clinical validity, or scientific benefit.

Conflict Status

Flag: a raw client cannot safely inherit arbitrary caller-selected endpoint behavior without potentially routing a credential to an unreviewed destination. Respond: Gate 0 keeps legacy flexibility confined to the SDK lane and makes the raw lane profile-bound. Decide: the PI retains future authority over live endpoint profiles, credential custody, and any compatibility expansion. Document: this record preserves the resulting offline boundary and its limits.

Execution Decision

Accept Gates 0 and 1 as a reversible offline foundation. Proceed only to a separately reviewed transport increment that consumes this contract, proves its behavior with synthetic inputs, and preserves the SDK compatibility lane until live and scientific gates are independently satisfied.
