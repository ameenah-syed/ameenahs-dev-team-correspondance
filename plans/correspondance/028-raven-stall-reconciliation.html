Confirmed Root Causes

  1. The exact apply_patch failures are Windows restricted-token sandbox-wrapper preparation failures caused by an unenforceable split writable-root set. They occur before target file access.
  2. Shared subagents inherit the parent task's cwd/root advertisement; pointing them at a nested worktree does not create an exact-worktree sandbox.
  3. Some formatter/test waits are separate subprocess and resource-contention failures. Outer tool termination may leave child Python processes alive.

Contributing Factors

  • Parent-repository cwd differs from the target Git worktree.
  • Coverage-enabled broad pytest defaults and no command watchdog increase latency and silence.
  • Short 10-second probes can misclassify normal Windows startup, while broad recursive probes can become part of the problem.
  • Fatal temp or multiprocessing permission errors can fail tests; .pytest_cache warnings are secondary unless an exact fatal exception proves otherwise.
  • Trusting UI running without new execution evidence extends stalls.

Why Relocation Helped but Did Not Solve It

Moving from C:\tmp to <repo>\.worktrees fixed path containment and let direct shell/Git operations reach targets without crossing the project tree. It did not change a shared task's cwd or collapse the repository, visualization, and temporary writable roots advertised to the Windows wrapper. Nesting is necessary hygiene, not sufficient sandbox isolation.

Durable Raven Workflow

  1. Preflight: record thread, owner, exact cwd, git rev-parse --show-toplevel, branch, canonical target, declared writable roots, dirty baseline, exact command, timeout, and integration owner.
  2. For direct editing, launch a dedicated Codex task whose cwd and writable workspace root are exactly the issue worktree. Do not infer this from nesting.
  3. If a shared agent hits wrapper refusal, allow one focused recovery. If exact-root routing is unavailable or also fails, stop retries and require a complete unified diff. Raven stores it in the repository, runs git apply --check, applies it, inspects status/diff, and routes independent verification.
  4. For pytest, start with one exact node using python -B -m pytest <node> -q --maxfail=1 --no-cov -p no:cacheprovider and an explicit 60–120 second bound. Use a unique in-worktree --basetemp only when temp permissions are implicated. Avoid parallelism and broad suites until the baseline passes.
  5. For formatters, target named files with check-only mode. Confirm Node/npm availability before npm work. Capture and terminate only the owned process tree when a timeout occurs.
  6. At 30 seconds with no output, inspect output time, Git status/diff, timestamps, and owned process CPU. At a second consecutive 30-second checkpoint with no new evidence, classify stalled regardless of UI state, interrupt before reassignment, and preserve partial work. If evidence advances, allow work only to the declared budget.
