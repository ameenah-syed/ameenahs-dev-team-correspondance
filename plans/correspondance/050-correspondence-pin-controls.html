Request Summary

The PI requested a real pinning feature for the correspondence website, including a small pin button in the interface. The existing fixed-top correspondence behavior had to become a removable pin rather than immutable archive data.

Desired outcome: visitors can pin and unpin any correspondence, see pinned records first, and keep that preference across reloads in the same browser. Non-goals: no login, account synchronization, server database, private-data publication, redesign, correspondence renumbering, or content rewrite.

Constraints and approval: PI-approved implementation and publication; one editor at a time; independent review and QA; dependency-free static GitHub Pages architecture; preserve every existing record and make infrastructure failures explicit.

Execution Update

The archive now maintains a browser-local set of pinned correspondence IDs. Records 045 and 048, which previously used permanent pinned:true flags, now use seed-only initiallyPinned:true metadata. On a visitor's first load, those IDs initialize the local pin set. Once the visitor saves any choice, including an empty set, the seed metadata no longer forces either record back to the top.

  • Every archive card has a real pin button placed outside its navigation link.
  • Every detail page has the same pin/unpin control.
  • Buttons expose dynamic aria-pressed, label, title, keyboard focus, and active styling.
  • Multiple pins sort first; each group retains descending date and ID ordering.
  • Archive search text survives a pin-triggered redraw and focus moves to the replacement button.
  • Malformed or unavailable storage cannot prevent rendering; current-page toggling remains available.
  • A storage event updates another open tab without changing the archive's route contract.
  • Mobile styling reserves room for the control and visually hides only its short text label, not its accessible name.

Scope Limits

This feature is intentionally local to each browser and device. It does not synchronize pins between computers, browsers, or signed-in identities because the static site has no account or backend service. Clearing this site's browser storage resets the archive to its two initial pins.

Live Edge/Chrome visual interaction remained unavailable in the current task. The browser automation package was missing a required sibling; its bounded installation stopped on the organization's certificate issuer validation. Native browser child startup was then denied by the current medium-integrity filtered process. TLS verification and browser sandboxing were not weakened. This is a known visual-verification gap, not evidence of an application defect.

Conflict Status

Flag: the old data model made records permanently top-ranked, while the PI requested removable user pins. Two legacy records, not one, were found with hard-coded pin flags.

Respond: silently dropping either legacy flag would change the existing first-load archive state. Keeping the old flags in sorting would prevent real unpinning.

Decide: migrate every legacy hard pin into first-load seed metadata, then make browser storage the sole active pin authority.

Document: this record preserves the two-record discovery, the migration rule, and the per-browser boundary.

Execution Decision

Disposition: Accept. Raven accepted the reviewed implementation for publication. Browser-local persistence is the smallest solution compatible with a dependency-free static website. A synchronized account-level pin system is deliberately deferred unless the PI later requests a backend and identity model.
