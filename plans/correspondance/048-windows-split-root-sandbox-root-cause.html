Request Summary

The PI requested an evidence-backed review of OpenAI Codex issues #10601 and #30009, their linked records, and current source to identify why native Windows sandbox preparation blocks apply_patch.

Desired outcome: distinguish the confirmed defect from adjacent failures, workarounds, and unsafe privilege broadening. Non-goals: no released-fix claim, local Codex modification, or recommendation to disable sandbox controls.

Execution Update

The split-root error occurs before patch parsing or target mutation. Codex Desktop supplies a managed filesystem profile containing several writable roots, commonly the workspace and temporary/runtime locations. The Windows unelevated restricted-token path compares those requested roots with its older compatibility workspace-write projection. When the sets differ, it refuses to run because the backend does not currently forward and enforce the complete split-root policy.

The refusal safely prevents silent unrestricted execution, but makes a valid managed profile unusable. Reports where Add succeeds while Update or Delete fails show execution-path asymmetry: existing-file operations enter the sandbox preparation/read path that performs the unsupported-policy check.

Issue #10601 is a related but broader setup-refresh family with competing explanations involving mutex concurrency, ACL inspection, SID handling, file locks, and refresh coverage. It is not proof that every Windows sandbox failure has one cause.

Scope Limits

The investigation used public OpenAI issues, pull requests, source, architecture material, and actual bounded responses from four assigned standing-role reviewers. It did not inspect proprietary Desktop host code, establish contributor proposals as maintainer determinations, or verify a released binary containing a complete correction.

Evidence

  • Issue #30009 records the exact pre-execution split writable-root refusal.
  • Issue #30712 independently demonstrates workspace-plus-temp roots producing the same failure before an existing file is patched.
  • Current Windows policy-resolution source contains the exact comparison and refusal.
  • PR #10648 combined SID, mutex, setup, keyring, and unrelated changes; it closed without merge.
  • Merged PR #31138 repairs deletion rights within writable roots but expressly excludes split-root setup, elevated-helper, and proxy failures.
  • OpenAI's Windows sandbox engineering description explains the restricted-SID and ACL model a complete fix must preserve.

Conflict Status

Flag: several plausible causes and successful elevated runs can be mistaken for proof of one repair.

Respond: separate #10601's setup-refresh family from #30009/#30712's exact split-policy refusal; classify elevation, PowerShell rewrites, git apply, and root removal only as workarounds.

Decide: accept the source-confirmed root-set comparison as the direct cause of the named error while retaining adjacent helper failures as separate unresolved classes.

Document: this record states the implementation contract without claiming a released fix.

Execution Decision

Disposition: require a launcher-level multi-root correction. The unelevated backend must canonicalize and deduplicate every effective Windows root, pass the complete write-root override through the launcher, apply path-scoped capability and ACL grants to every declared root, preserve protected/read-only carve-outs, validate all grants before child creation, and roll back partial setup failures. It must never widen access to a common ancestor, retain only the first root, or fall through unsandboxed.

A temporary single-root profile is only a reduced-capability workaround. Excluding temporary roots can break compilers, tests, installers, and other tools. Elevation may enter another backend but does not repair the unelevated policy model.
