Request Summary

The PI requested a plain-language explanation of why some engineering teams replace provider SDKs with direct HTTP clients, what the current ICVision SDK path does, and what a small project-owned HTTP transport would need to take over.

Desired outcome: support an informed future decision about ICVision’s provider boundary. Non-goals: no model selection, endpoint-policy decision, code change, credential handling, real API call, scientific claim, or release decision.

Execution Update

Current ICVision is an application pipeline around a provider SDK. It prepares a temporary component image, encodes that image as text-safe Base64, supplies the classification prompt and model settings, and calls the provider’s Responses-style API through the SDK. The SDK then owns low-level request construction, HTTPS communication, authentication-header handling, response-object conversion, and provider-specific error types. ICVision parses the returned classification text, validates it, and produces its usual result row.

A raw HTTP transport would remove only the SDK layer. ICVision would still prepare the image, choose the prompt and model, validate the classification, and preserve existing result semantics. The project would additionally own the exact request JSON, HTTPS destination, TLS behavior, headers, timeout, retry policy, response-size limit, response parsing, error redaction, and tests.

Why a thin HTTP transport can be useful

  • It makes the outbound contract visible and testable: destination, headers, JSON fields, storage request, bounds, and error handling are project code rather than implicit library behavior.
  • It can reduce dependency and upgrade churn when the pipeline uses a small, stable subset of one API.
  • It can support a documented provider-profile boundary rather than coupling ICVision internals to one SDK’s object model.
  • It can make synthetic testing more direct: a mock connection can prove that no network call occurs and that sensitive provider text never reaches retained logs.

These benefits do not mean that an SDK is automatically unsafe or obsolete. OpenAI documents both its official SDKs and use of a developer’s own HTTP client. The tradeoff is control versus maintenance responsibility.

Evidence

  • Read-only inspection of ICVision’s application flow found an SDK-created Responses request after image preparation and before application-level classification parsing.
  • A reviewed raw-transport candidate demonstrates bounded image/request/response handling, explicit store: false, certificate-verifying HTTPS, and sanitized failure text, but it is not integrated into the active workflow.
  • OpenAI’s current documentation presents official SDKs and a developer’s own preferred HTTP client as supported ways to make direct API requests: SDKs and CLI.

Conflict Status

Flag: ICVision’s existing compatibility surface can accept a custom API base URL, while a fixed or profile-based raw client should not silently route a credential to an arbitrary host.

Respond: preserve current behavior during a staged migration; do not treat a raw client as a drop-in replacement until its endpoint and credential policy is explicit.

Decide: pending PI decision. The proposed direction is to preserve the current SDK compatibility lane initially and make any new raw transport opt-in with named, reviewed endpoint profiles.

Document: this record captures the tradeoff; it does not choose the final policy.

Execution Decision

Disposition: Advisory only; no architecture selection. The current SDK implementation is the existing compatibility reference and is not approval for live provider use. If the PI later authorizes a raw transport, it should be deliberately small and first prove request safety, output compatibility, and rollback behavior behind an opt-in seam. Removing the SDK or accepting arbitrary custom destinations in a raw client requires a later PI-approved compatibility decision.
