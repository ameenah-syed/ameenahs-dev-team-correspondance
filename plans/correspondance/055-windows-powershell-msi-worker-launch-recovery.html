Request Summary

Recover normal Codex worker-task launch after repeated pre-child CreateProcessAsUserW failed: 5 failures. No UAC-policy, registry-security, WindowsApps-permission, Codex-source, credential, data, or ICVision production-code change was authorized.

Execution Update

The Microsoft Store/MSIX PowerShell alias was the practical executable-resolution problem. The official PowerShell 7.6.4 x64 MSI was downloaded, SHA-256 verified, installed, placed first in user PATH, and followed by a full Codex restart. A fresh normal sandboxed Cricket task read required instructions and ran Get-Location with exit code 0.

Scope Limits

This proves recovery of the observed worker-launch symptom only. It does not prove that the Codex restricted-token or multi-root sandbox-policy defect, every role, patch helper, network feature, or ICVision test is healthy.

Evidence

  • Official MSI SHA-256: D11942DF52FD12470169797ABFA4781D9480EFDC81000BA4FA55A5B921ED8DD0.
  • Installer exit code 0; Program Files PowerShell exists and reports 7.6.4.
  • After restart, the coordinator resolved pwsh to the Program Files executable.
  • Fresh Cricket launch check exited 0 without writes, credentials, data, or network access.

Conflict Status

Executable resolution is not a platform sandbox-policy fix. This is accepted only as an operational recovery while the upstream diagnosis remains open.

Execution Decision

Use the restored normal worker path for scoped ICVision work. Do not weaken sandbox boundaries or claim the platform issue is closed.

Verification

Cricket independently read project instructions and ran Get-Location in a fresh normal sandboxed task; it passed with exit code 0 at the intended ICVision cwd.

Next Actions
