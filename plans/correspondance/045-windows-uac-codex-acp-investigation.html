Request Summary

The PI requested a durable account of the permission investigation that began while testing agentclientprotocol/codex-acp as a possible way to reduce recurring task-launch and coordination friction. The desired outcome is a frequently reusable answer to three questions: what the proof of concept actually established, why an Administrator-group account can still report a non-administrator process, and which recovery actions are safe. The PI explicitly authorized public publication and top-of-site pinning.

Non-goals were disabling Windows security controls, inspecting credentials, calling a live model or API, moving protected data, modifying upstream Codex, claiming an ACP adapter defect without evidence, or representing process elevation as the platform fix for split writable roots.

Execution Update

  1. A bounded synthetic ACP proof of concept was selected: read-only agent mode, no project-file edits, no network/API operation, and no external data.
  2. The required Node runtime launched successfully. The adapter then attempted its documented next step: spawning the bundled Codex App Server.
  3. That child-process creation failed with Windows spawn EPERM inside the restricted tool sandbox. The failure occurred at a permission boundary after Node startup and before a usable end-to-end ACP session.
  4. The investigation then separated Windows account membership from the token on the current process. Under UAC, an Administrator-group account normally starts ordinary applications with a filtered standard token. A separate elevated token is used only when a new process is launched with approval.
  5. Restarting Windows did not make an already ordinary Codex launch elevated. A token is established when the process starts; it is not manually edited in place.
  6. The safe operator action was therefore narrowed to temporary, explicit elevation of a new process only when the operation truly requires it.

Supported temporary elevation example:

Start-Process powershell.exe -Verb RunAs

Verify the newly opened process, not the old one:

$id = [Security.Principal.WindowsIdentity]::GetCurrent()
$p = [Security.Principal.WindowsPrincipal]::new($id)
$p.IsInRole([Security.Principal.WindowsBuiltInRole]::Administrator)

For a desktop application, close all existing application processes and explicitly start a new instance with Run as administrator. End the elevated session after the bounded operation.

Scope Limits

The recorded proof did not complete ACP initialization, exercise protocol requests, modify a repository, invoke a provider, inspect credentials, or transfer data. No claim is made that temporary elevation completed the later ACP rerun because no exact completion evidence was available in the originating record at publication time. The test did not diagnose every possible source of Windows EPERM; it classified the exact observed boundary.

Evidence

  • Observed: Node launched; codex-acp then received spawn EPERM while starting its bundled Codex App Server inside the restricted tool sandbox.
  • Observed: the process-level administrator test remained false after a reboot because Codex was not started as a newly elevated process.
  • Authoritative Windows behavior: Microsoft documents that Administrator-group accounts receive standard and administrator tokens and that ordinary scripts use the filtered token until run elevated.
  • Adapter contract: the agentclientprotocol/codex-acp README states that the stdio ACP server starts Codex App Server and supports a read-only initial mode.
  • Prior diagnosis: ICVision correspondence 008 classifies the split-root refusal at the task-launch/sandbox-policy boundary; correspondence 009 defines the least-privilege platform remediation and acceptance tests.

References: Microsoft: User Account Control and WMI; Microsoft: Start-Process; agentclientprotocol/codex-acp.

Conflict Status

Flag: temporary admin access was initially expected to eliminate all Windows permission errors. Respond: separate filtered-token behavior, genuine administrator prerequisites, ACP child-spawn policy, and the known split-root launcher defect. Decide: permit bounded elevation for genuine admin-required operations, but never call elevation the sandbox root-cause fix. Document: this record and cross-linked correspondences 008 and 009.

Execution Decision

Modify and accept. Use dependency-first recovery and temporary per-process elevation where justified. For ACP, retain the smallest synthetic read-only proof with a clear stop condition. For split writable roots, keep the platform issue open until ordinary sandboxed tools can operate in every declared root while denying outside-root access. Raven owns coordination; the PI retains authority for security-boundary changes and external operations.

Publication Completeness Addendum

Desired Outcome and Approval

A frequently reusable, top-pinned GitHub Pages reference was explicitly requested and approved by the PI.

Assumptions and Alternatives

The diagnosis assumes the recorded EPERM occurred at child spawn as observed. Alternatives such as adapter logic failure remain unevaluated because the child never started. Safe alternatives are a properly authorized bounded rerun or continued use of existing Codex task routing; neither changes the split-root platform requirement.

Custody and Redaction

The originating ICVision HTML is the canonical detailed record. The public archive copy omits machine identity, sensitive paths, credentials, raw output, and protected data.

Known Gaps and Stop Conditions

Stop after one repeated failure of the same spawn class, any request for credentials outside the approved auth flow, any attempted network/API action, any project-file mutation, or any pressure to weaken UAC or sandbox enforcement. A future successful rerun should be documented in a new addendum.
